Tools and Technologies

For our project, there are tools and technologies that are split between developers and end users. For developers of this project, there are three main technologies that will be used which are QR Codes, NFC Tags and Progressive Web Applications. Lets dive a bit deeper into the three technologies.

QR Codes

To start off, everyone has probably heard of QR Codes, and in these modern pandemic times, everyone has also hopefully scanned at least one as well. In essence, QR codes is an advanced barcode that comes in the form of a square. It is capable of storing larger amounts of information within a combination of pixels in a square grid. It can also be generated very easily at no cost and as such using QR codes for our project would be very feasible. It can also be read and interpreted quite easily with any modern-ish mobile device, making it a simple process for end users.

NFC Tags

Next we have NFC tags. Although not as prominent as QR codes, NFC tags are still also widely used in society. NFC tags essentially allow communication between a tag and reader that are within close proximity of each other. The use of these tags are more convenient and versatile as compared to QR codes, as all it requires is a tap on a device compared to having to open up an app and manually scan the QR code. NFC Tags also can be programmed quite easily with the use of an Android or iOS device. As such, using NFC tags for this project can be a more secure option, however it costs money (although not very expensive).

Progressive Web Applications

The use of QR codes and NFC tags can pose large security issues. Who is to say that a random person with malicious intent cannot just generate a random QR code, posing as one that is generated as part of our game, and have it linked to some sort of malware. Furthermore, in the same sense a random person can also program their own NFC tags once again disguised as one created from our game, and have it steal the end users personal information. This is where progressive web applications come into play. PWAs are a cross between a web and native installed application. Because of that, it takes many advantages from both, and mashes them together to create a better application. To address the security issue, we use PWAs because it enforces the HTTPs web protocol which ensures that any data stored on the application is secure and not tampered with. Also when using PWA, end users are ensured that they are accessing the real game, and not something malicious generated by a random person. As developers, we will be using Bootstrap Studio 6.0.1 to create the PWA which can then be accessed through a web browser, or a mobile application.

Testing

There are three different ways to test an application, which is manual testing, automated testing, and continuous testing. In our project, we will be using manual testing to test our project. It takes more time than automated tools but it’s more accurate than automated tools, it provides human insight, adaptable and save money. In manual testing, the testing process starts with understanding the application that's going to be tested. We will be gathering information about the application and the expectation of the users. When the data is collected, we will integrate the data and save it for later testing. Next steps is to create test scenarios and test cases, We will use the previously collected user needs and expectations to analyze and create a test that meets the requirements. Defects found during testing are recorded and fixed while testing is in progress. The purpose of testing is not only to identifying and reporting defects found, but also to ensure that all reported issues are identified and fixed, and then retested to confirm. After all defects have been eliminated, the acceptance testing method will be used to make sure our apparition is ready to go.

There will be four types of tests to be used to test our project.

- White Box Testing: for testing the behaviour of each of the features in the application. Is a method that allows testers to examine its code, infrastructure, and integration with external systems.

- Black Box Testing: Use it to test user interface, input, and output. It only considers the external behaviour of the system.

- Grey Box Testing: Is a method used to perform testing, risk, and assessment methods.

- Acceptance Testing: It is a software testing method in which the acceptability of a system is tested. The main purpose of this test is to assess whether the system meets the requirements and whether it can be delivered for use.

Risks

What risks can you identify for your project? There will always be some generic risks (such as computers breaking down the night before a deadline, health and family issues, and institutional changes). Do not include generic risks such as these. The idea is to be as specific as you can to your project. For example, if your topic is to develop a game, there may be a risk that the software you choose to work with may be very difficult to learn, poorly documented, or not turn out to have the features that it claims it has. These properties are often only discovered once you have started working with the software, and so unless you have had lots of experience with the particular tool, there is always a risk that it may not work as well as you believe it should, no matter how much prior research you do. Similar comments apply to hardware.
